# Workflow Policy Artifact Metadata Contract

This document freezes the workflow policy proof-of-work artifact metadata contract
for agent-taskflow. It defines stable canonical names and structures for artifact
filenames, store artifact types, review evidence file kinds, and package metadata.

This is a metadata contract freeze. It does not add:
- dispatcher runtime behavior or enforcement
- executor behavior changes
- validator registry changes
- API endpoints
- Mission Control frontend changes
- GitHub sync or PR creation
- merge, push, cleanup, or delete automation
- AI self-governance

## Canonical Artifact Filenames

These are the canonical filenames for workflow policy proof-of-work artifacts.
Future phases must not rename these files casually.

| Filename | Purpose |
|----------|---------|
| `workflow_policy_summary.json` | Machine-readable workflow policy summary artifact. |
| `artifact_index.json` | Package-level index listing all artifacts in the proof-of-work package. |

## Canonical Store Artifact Types

These are the accepted values for `artifact_type` when calling
`TaskMirrorStore.record_task_artifact()` for workflow policy artifacts.

| Artifact Type | Description |
|---------------|-------------|
| `workflow_policy_summary` | The workflow policy summary artifact file. |
| `artifact_index` | The proof-of-work package artifact index file. |

The base `TASK_ARTIFACT_TYPES` enum (in `agent_taskflow/models.py`) includes these
types. Existing `other` artifacts remain supported for backward compatibility. New
workflow policy artifacts should use these explicit types.

These store artifact types are used for metadata recording in the SQLite store.
They do not imply dispatcher enforcement or runtime policy checks.

## Canonical Review Evidence Kind

The review evidence helper (`agent_taskflow/api/review.py`) classifies artifact
files by name. The canonical proof-of-work artifact files are classified as:

| File Kind | Filename Pattern |
|-----------|------------------|
| `workflow_policy` | `workflow_policy_summary.json`, `artifact_index.json` |

This classification is read-only. It is used by `build_artifact_file_summaries()`
to assign a `kind` field to each artifact in the review evidence response.

The `kind` value `workflow_policy` is file-name based for these canonical filenames.
It does not add API endpoints or Mission Control UI behavior.

## Canonical Proof-of-Work Package Type

The `artifact_index.json` file includes a `package_type` field. The canonical
package type for workflow policy proof-of-work packages is:

| Package Type | Description |
|--------------|-------------|
| `workflow_policy_proof_of_work` | A proof-of-work package containing workflow policy artifacts. |

## artifact_index.json Contract

The `artifact_index.json` file is written by proof-of-work package generators
(e.g., `scripts/run_workflow_policy_pow_package_smoke.py`).

### Required Top-Level Fields

| Field | Type | Description |
|-------|------|-------------|
| `artifact_index_version` | string | Version of this index format. Currently `0.1`. |
| `package_type` | string | Must be `workflow_policy_proof_of_work`. |
| `generated_at` | string | ISO-8601 timestamp of package generation. |
| `artifacts` | list | List of artifact entries. |

### Artifact Entry Structure

Each entry in `artifacts` must be a dict with the following fields:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | yes | Human-readable artifact name. |
| `artifact_type` | string | yes | Store artifact type, e.g., `workflow_policy_summary`. |
| `path` | string | yes | Filename relative to the artifact directory. |
| `required` | boolean | yes | Whether this artifact is required for the package. |
| `description` | string | no | Human-readable description of the artifact. |

### Required Artifact Entry

Every `artifact_index.json` for a workflow policy proof-of-work package must
include this entry:

```json
{
  "name": "workflow_policy_summary",
  "artifact_type": "workflow_policy_summary",
  "path": "workflow_policy_summary.json",
  "required": true,
  "description": "Machine-readable workflow policy summary artifact."
}
```

## workflow_policy_summary.json Contract

The `workflow_policy_summary.json` file is generated by the artifact writer
(`scripts/write_workflow_policy_summary_artifact.py`) and contains a structured
summary of the workflow policy.

### Required Fields

| Field | Type | Description |
|-------|------|-------------|
| `artifact_type` | string | Must be `workflow_policy_summary`. |
| `schema_version` | string | Schema version of the summary (from policy `schema_version`). |
| `source_path` | string | Absolute path to the source policy JSON file. |
| `validation_status` | string | `passed` or `failed`. |
| `allowed_executors` | list[string] | Executors permitted by the policy. |
| `required_validators` | list[string] | Validators required by the policy. |
| `path_policy` | object | Path policy section from the policy. |
| `workspace_policy` | object | Workspace policy section from the policy. |
| `proof_of_work` | object | Proof-of-work policy section from the policy. |
| `human_review` | object | Human review policy section from the policy. |
| `forbidden_actions` | list[string] | Actions forbidden by the policy. |
| `deferred_integrations` | list[string] | Integrations deferred by the policy. |
| `governance_invariants` | list[object] | Governance invariant checks from the policy. |
| `generated_at` | string | ISO-8601 timestamp of artifact generation. |

### Optional Fields

| Field | Type | Description |
|-------|------|-------------|
| `validation_errors` | list[string] | Validation error messages if `validation_status` is `failed`. |
| `validation_warnings` | list[string] | Validation warning messages. |
| `optional_validators` | list[string] | Optional validators (if policy specifies them). |

### Example Structure

```json
{
  "artifact_type": "workflow_policy_summary",
  "schema_version": "0.1",
  "source_path": "/path/to/workflow-policy.example.json",
  "validation_status": "passed",
  "validation_errors": [],
  "validation_warnings": [],
  "allowed_executors": ["manual", "shell", "opencode", "pi"],
  "required_validators": ["policy", "changed-files", "pytest", "typecheck", "lint"],
  "optional_validators": ["openspec"],
  "path_policy": {
    "allowed_paths": [],
    "forbidden_paths": []
  },
  "workspace_policy": {
    "isolation_required": true,
    "preferred_strategy": "per_task_worktree",
    "preserve_on_failure": true,
    "cleanup_control": "human_or_deterministic_policy"
  },
  "proof_of_work": {
    "required_artifacts": ["run_summary", "mission_contract"],
    "optional_artifacts": ["artifact_index", "handoff_decision"]
  },
  "human_review": {
    "required": true,
    "allowed_decisions": ["approve", "reject", "rerun", "block"]
  },
  "forbidden_actions": ["self_approve", "approve_without_human", "push"],
  "deferred_integrations": ["github_issues_sync", "automatic_pr_creation"],
  "governance_invariants": [
    {
      "invariant": "ai_workers_may_approve",
      "value": false,
      "description": "AI workers may not approve their own tasks."
    }
  ],
  "generated_at": "2025-01-01T00:00:00Z"
}
```

## Stability Rule

Future phases must not rename canonical filenames, artifact types, file kinds, or
top-level fields without:

1. A compatibility note in this document.
2. Tests covering the renamed fields.
3. A deprecation period or migration path for existing data.

API and Mission Control UI layers should consume these names as read-only stable
identifiers. Runtime enforcement should not infer more than the contract states.

## Non-Goals

This contract does not add:
- **Dispatcher enforcement**: The artifact metadata is proof-of-work evidence.
  It does not trigger dispatcher behavior or runtime validation enforcement.
- **Executor behavior changes**: Executors are unaffected by artifact type
  additions.
- **Validator registry changes**: The workflow policy artifact metadata does not
  alter which validators run or how.
- **API endpoints**: No new endpoints are added or modified.
- **Mission Control UI**: No frontend changes are introduced by this contract.
- **GitHub sync or PR creation**: Artifact types do not trigger GitHub operations.
- **Merge, push, cleanup, or delete automation**: These actions remain governed
  by the `forbidden_actions` list in the policy, not by artifact metadata.
- **AI self-governance**: AI workers receive policy context but do not enforce it.

## Migration and Compatibility

- **Existing "other" artifacts** remain valid. The store accepts `other` as a
  backward-compatible artifact type.
- **New workflow policy artifacts** should use `workflow_policy_summary` and
  `artifact_index` types going forward.
- **DB schema is unchanged**: The artifact type is stored as TEXT in the
  `task_artifacts` table. No migration is required for adding new types to
  `TASK_ARTIFACT_TYPES`.
- **Review evidence kind** is file-name based for canonical filenames. The
  `workflow_policy` kind is assigned when the filename matches
  `workflow_policy_summary.json` or `artifact_index.json`. This is stable
  for the canonical filenames defined above.

## Reference Implementation

The canonical implementation of artifact generation and verification is:

- `scripts/write_workflow_policy_summary_artifact.py` — generates
  `workflow_policy_summary.json` using `build_artifact()`.
- `scripts/run_workflow_policy_pow_package_smoke.py` — generates the complete
  proof-of-work package including `artifact_index.json`.
- `scripts/run_workflow_policy_review_evidence_smoke.py` — verifies artifact
  metadata through the store API and review evidence helper.
- `agent_taskflow/api/review.py` — classifies artifact files by name using
  `_WORKFLOW_POLICY_ARTIFACT_NAMES` and `_file_kind()`.
- `agent_taskflow/models.py` — defines `TASK_ARTIFACT_TYPES` including
  `workflow_policy_summary` and `artifact_index`.

## Changes to This Document

Any addition or change to this contract should update the reference implementation
and add tests. If a new canonical filename or artifact type is added, the Phase
number and description of the addition should be recorded at the top of this
document.

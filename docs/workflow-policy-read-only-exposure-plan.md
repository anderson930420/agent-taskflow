# Workflow Policy Read-Only Exposure Plan

## Purpose

This document defines how workflow policy proof-of-work artifacts may eventually be
exposed read-only through the review evidence API, Mission Control UI, and future
read-only endpoints.

**This phase does not implement any API, UI, dispatcher, or runtime behavior.** It
is a planning and alignment document for future phases.

The canonical artifact metadata contract (docs/workflow-policy-artifact-metadata-contract.md)
defines stable names and shapes. This document describes how those artifacts may be
displayed read-only, without implying runtime enforcement.

---

## Current Artifact Chain

The workflow policy proof-of-work artifact chain is fully implemented as standalone
scripts with tests:

```
workflow-policy.example.json
  → workflow schema loader (agent_taskflow/workflow_schema.py)
  → workflow policy validation command (scripts/validate_workflow_policy.py)
  → workflow policy summary command (scripts/summarize_workflow_policy.py)
  → workflow_policy_summary.json artifact generator
      (scripts/write_workflow_policy_summary_artifact.py,
       agent_taskflow/workflow_policy_artifacts.py)
  → workflow policy artifact smoke (scripts/run_workflow_policy_artifact_smoke.py)
  → workflow policy proof-of-work package smoke
      (scripts/run_workflow_policy_pow_package_smoke.py)
  → workflow policy review-evidence smoke
      (scripts/run_workflow_policy_review_evidence_smoke.py)
```

The canonical artifact metadata contract adds these stable identifiers:

| Element | Canonical value |
|---------|-----------------|
| Summary filename | `workflow_policy_summary.json` |
| Index filename | `artifact_index.json` |
| Summary artifact type | `workflow_policy_summary` |
| Index artifact type | `artifact_index` |
| Review evidence kind | `workflow_policy` |
| Package type | `workflow_policy_proof_of_work` |
| Artifact index version | `0.1` |

These artifacts are stored in the task artifact directory and recorded via the
`TaskMirrorStore.record_task_artifact()` API. They are not runtime enforcement
controls.

## Canonical Artifacts

### `workflow_policy_summary.json`

A machine-readable summary of the workflow policy at the time of artifact generation.
Generated by `scripts/write_workflow_policy_summary_artifact.py` using
`build_artifact()` from the policy loader.

Required fields (from `WORKFLOW_POLICY_REQUIRED_SUMMARY_FIELDS`):

| Field | Description |
|-------|-------------|
| `artifact_type` | Must be `workflow_policy_summary` |
| `schema_version` | Schema version from the policy |
| `source_path` | Absolute path to the source policy JSON |
| `validation_status` | `passed` or `failed` |
| `validation_errors` | List of error messages (required, may be empty) |
| `validation_warnings` | List of warning messages (required, may be empty) |
| `allowed_executors` | List of permitted executor names |
| `required_validators` | List of required validator names |
| `path_policy` | Path allow/forbid lists |
| `workspace_policy` | Workspace isolation and cleanup settings |
| `proof_of_work` | Required and optional proof-of-work artifacts |
| `human_review` | Human review policy |
| `forbidden_actions` | Actions forbidden by the policy |
| `deferred_integrations` | Integrations deferred by the policy |
| `governance_invariants` | Governance invariant checks |
| `generated_at` | ISO-8601 timestamp of generation |

`optional_validators` may be present but is not a required field.

### `artifact_index.json`

A package-level index listing all artifacts in the proof-of-work package.
Generated by `scripts/run_workflow_policy_pow_package_smoke.py`.

Required top-level fields (from `WORKFLOW_POLICY_REQUIRED_INDEX_TOP_LEVEL_FIELDS`):

| Field | Description |
|-------|-------------|
| `artifact_index_version` | Format version (`0.1`) |
| `package_type` | Must be `workflow_policy_proof_of_work` |
| `generated_at` | ISO-8601 timestamp of generation |
| `artifacts` | List of artifact entries |

Each entry in `artifacts` must include: `name`, `artifact_type`, `path`, `required`.

## Read-Only Exposure Goals

The following display goals are future intentions, not current behavior.
They guide API and UI design but require separate implementation phases:

- Show which workflow policy artifact package is associated with a task or run
- Show policy `validation_status` (passed/failed)
- Show policy `schema_version`
- Show `allowed_executors`
- Show `required_validators`
- Show path policy summary (allowed/forbidden paths)
- Show workspace policy summary (isolation strategy, cleanup control)
- Show proof-of-work artifact requirements
- Show human review policy (required, allowed decisions)
- Show `forbidden_actions`
- Show `deferred_integrations`
- Show `generated_at` / `source_path`

These goals are read-only. Displaying policy metadata does not trigger
dispatcher behavior, executor changes, or approval automation.

## Proposed Staged Exposure Path

### Stage 1 — Standalone scripts and artifact package

**Status: implemented.**

Proof-of-work artifacts are generated by standalone scripts. They are
recorded via the store API and exposed through existing review evidence.
No API shape change, no dispatcher automation, no UI.

### Stage 2 — Controlled review evidence attachment

**Status: implemented.**

The `run_workflow_policy_review_evidence_smoke.py` script exercises the
full artifact chain: policy loading → artifact generation → store recording
→ review evidence retrieval via the API.

Future work at this stage may include:

- Ensuring workflow policy artifacts are always recorded in the store
- Verifying `kind == "workflow_policy"` is assigned to both canonical filenames
- Ensuring review evidence includes `workflow_policy_summary.json` and
  `artifact_index.json` in the artifact list

No dispatcher automation is added at this stage.

### Stage 3 — Read-Only API Exposure

**Status: not implemented. Requires design and tests.**

Expose workflow policy artifacts through the existing review evidence
response or a clearly read-only field (e.g., `workflow_policy_summary` in
the review evidence response top-level).

Requirements:

- No state transition changes
- No enforcement claim
- Read-only: GET endpoint only
- Follows existing response shape conventions
- Covered by contract tests before shipping

### Stage 4 — Mission Control Read-Only Display

**Status: not implemented. Requires design and tests.**

Display workflow policy metadata in review/evidence panels.

Requirements:

- Display only; no buttons
- No approve/merge/push/cleanup behavior
- Reads from review evidence API
- Covered by UI doc tests before shipping

### Stage 5 — Dispatcher Preflight Planning

**Status: not planned. Requires separate design and tests.**

Dispatcher preflight may check that required workflow policy proof-of-work
artifacts are present before certain operations. This is a separate,
later phase.

Preconditions for Stage 5:

- Stages 1–4 are stable and tested
- A separate preflight design document exists
- Contract tests lock the preflight behavior
- Human review remains the final gate

## Safety Wording

The following principles are non-negotiable for all future exposure work:

1. **Display is not enforcement.** Showing workflow policy metadata in a UI
   or API response does not imply that the dispatcher enforces that policy.
   Enforcement is handled by validators and deterministic code.

2. **Read-only evidence only.** Workflow policy artifacts are proof-of-work
   metadata. They record policy state at generation time. They do not cause
   state transitions.

3. **Human review is the final gate.** Approval decisions remain with the
   human reviewer. Approval does not imply merge, push, or cleanup.

4. **Dispatcher enforcement is a later phase.** Any future dispatcher
   preflight or enforcement must be designed separately and tested against
   explicit contract tests.

5. **AI workers receive context, not authority.** AI workers may receive
   policy context (allowed executors, forbidden actions, etc.) to inform
   their work. They do not enforce policy or make governance decisions.

## Non-Goals

This exposure plan does not add:

- **API endpoint changes**: No new endpoints or modified existing ones.
- **Mission Control UI changes**: No frontend additions or modifications.
- **Dispatcher enforcement**: No runtime policy checks or preflight enforcement.
- **Executor behavior changes**: Executors are unaffected by artifact exposure.
- **Validator registry changes**: Validator selection is unchanged.
- **GitHub sync**: No GitHub API calls or repository state changes.
- **GitHub Projects integration**: No Projects board updates.
- **PR creation**: No pull request automation.
- **Automatic merge**: No merge automation.
- **Automatic push**: No push automation.
- **Automatic cleanup/delete**: No worktree cleanup or deletion automation.
- **AI self-governance**: AI workers do not approve, block, or enforce policy.
- **Prompt-only governance**: Policy context delivered via prompts is out of scope.

## Preconditions Before API/UI Work

Before read-only API exposure (Stage 3) or Mission Control UI display (Stage 4),
the following preconditions must be met:

1. `scripts/run_local_validation.py` passes all checks
2. Workflow policy artifact package contract tests pass
   (`tests.test_workflow_policy_artifact_package_contract`)
3. Review evidence `workflow_policy` kind tests pass
   (`tests.test_review_evidence_workflow_policy_artifacts`)
4. Artifact metadata constants contract tests pass
   (`tests.test_workflow_policy_artifact_constants_contract`)
5. Workflow policy review-evidence smoke passes
   (`scripts/run_workflow_policy_review_evidence_smoke.py`)
6. No dispatcher/runtime behavior changes in the same phase as API/UI work
7. No frontend write/action controls (approve/merge/push/cleanup buttons) in
   the same phase as read-only display

These preconditions ensure that the artifact chain is stable before exposure
is added. Adding exposure to an unstable artifact chain would risk displaying
stale or incorrect metadata.

## Recommended Next Phase

**Preferred: Phase 106 — Add read-only workflow policy artifact exposure contract
tests for the existing review evidence shape.**

Rationale: Adding contract tests first locks the expected review evidence response
shape before API or UI exposure is designed. This follows the established pattern
from Phases 102–104:

- Phase 102 locked doc ↔ constants alignment
- Phase 103 locked review evidence `_file_kind()` classification
- Phase 104 locked artifact package structure

Locking the expected response shape first means API and UI work in subsequent
phases has a stable target. It also prevents accidental shape drift.

**Alternative: Phase 106 — Add a standalone workflow policy review evidence report
command.**

This would add a script that reads the store and produces a human-readable report
of workflow policy artifacts for a task. Useful for standalone reporting without
requiring the API. Lower risk than API/UI changes.

The preferred option (exposure contract tests) is recommended because it provides
the most durable foundation for future Stages 3 and 4.
